QUESTION 4: "Can you list the names and USNs of students residing in Room 101?"
--------------------------------------------------------------------------------
- DBMS Concept: Multi-Table Join (Joining users, students, and rooms tables)
- SQL Query:
  SELECT u.name, s.usn, s.course, s.branch, r.room_number
  FROM students s
  JOIN users u ON s.user_id = u.id
  JOIN rooms r ON s.room_id = r.room_id
  WHERE r.room_number = '101';

- Expected Output:
  +----------+------------+--------+--------+-------------+
  | name     | usn        | course | branch | room_number |
  +----------+------------+--------+--------+-------------+
  | John Doe | 1RV22CS045 | B.Tech | CSE    | 101         |
  +----------+------------+--------+--------+-------------+


QUESTION 5: "Show me all students who have unpaid or partial room fees."
--------------------------------------------------------------------------------
- DBMS Concept: Financial Audit & Range Selection
- SQL Query:
  SELECT f.fee_id, u.name, f.amount, f.due_date, f.status

QUESTION 8: "How many unique students have filed at least one complaint?"
--------------------------------------------------------------------------------
- DBMS Concept: DISTINCT Keyword inside Aggregation
- Explanation: Without DISTINCT, if 'Student A' filed 5 complaints, they would be 
  counted 5 times. DISTINCT ensures we count unique individuals.
- SQL Query:
  SELECT COUNT(DISTINCT student_id) as unique_students_complained FROM complaints;

- Expected Output:

  | unique_students_complained  |
  +-----------------------------+
  | 14                          |
  +-----------------------------+

QUESTION 12: "How do you prove that your database has absolute Referential Integrity 
             and contains no orphaned student records (i.e. students without a user)?"
--------------------------------------------------------------------------------
- DBMS Concept: Outer Join (LEFT JOIN) filtering for NULL pointers
- Explanation: A LEFT JOIN yields all students and their matching users. If a student
  profile has NO user account, the 'users.id' will return NULL. Running this check 
  should return an empty set (0 rows), proving 100% Referential Integrity!
- SQL Query:
  SELECT s.student_id, s.usn 
  FROM students s 
  LEFT JOIN users u ON s.user_id = u.id 
  WHERE u.id IS NULL;

- Expected Output:
  Empty set (0.00 sec)  <-- (This is the perfect output, proving zero errors!)


QUESTION 13: "Write a Nested Subquery to find students with fees higher than the average fee."
--------------------------------------------------------------------------------
- DBMS Concept: Non-Correlated Subquery & Sub-selection comparison
- Explanation: The inner query '(SELECT AVG(amount) FROM fees)' returns a single numeric
  value (the average fee). The outer query then filters students whose fee exceeds this average.
- SQL Query:
  SELECT u.name, s.usn, f.amount
  FROM students s
  JOIN users u ON s.user_id = u.id
  JOIN fees f ON s.student_id = f.student_id
  WHERE f.amount > (SELECT AVG(amount) FROM fees);

- Expected Output:
  +-----------+------------+----------+
  | name      | usn        | amount   |
  +-----------+------------+----------+
  | John Doe  | 1RV22CS045 | 67000.00 |
  +-----------+------------+----------+


QUESTION 14: "Audit private messages showing both sender and recipient names."
--------------------------------------------------------------------------------
- DBMS Concept: Multi-Join on the Same Table (Double Join with distinct aliases)
- Explanation: The message table stores 'sender_id' and 'recipient_id', both referencing 
  'users.id'. We must join the 'users' table twice, aliasing one as 'sender' and one as 'recipient'.
- SQL Query:
  SELECT m.message_id, sender.name as sender_name, recipient.name as recipient_name, m.content 
  FROM private_messages m
  JOIN users sender ON m.sender_id = sender.id
  JOIN users recipient ON m.recipient_id = recipient.id
  ORDER BY m.created_at DESC;

- Expected Output:
  +------------+-------------+----------------+--------------------------+
  | message_id | sender_name | recipient_name | content                  |
  +------------+-------------+----------------+--------------------------+
  | 1          | John Doe    | Warden         | Can I ask about leaves?  |
  +------------+-------------+----------------+--------------------------+


QUESTION 15: "Show me which rooms currently have bed vacancies."
--------------------------------------------------------------------------------
- DBMS Concept: Direct attribute comparison (Integer logic operations)
- SQL Query:
  SELECT room_number, block, floor, capacity, occupied, (capacity - occupied) as vacant_beds 
  FROM rooms 
  WHERE occupied < capacity;

- Expected Output:
  +-------------+-------+-------+----------+----------+-------------+
  | room_number | block | floor | capacity | occupied | vacant_beds |
  +-------------+-------+-------+----------+----------+-------------+
  | 101         | A     | 1     | 3        | 1        | 2           |
  | 102         | A     | 1     | 3        | 0        | 3           |
  +-------------+-------+-------+----------+----------+-------------+


QUESTION 16: "What is the average outstanding fee balance grouped by study branch?"
--------------------------------------------------------------------------------
- DBMS Concept: Multi-Table Joins coupled with GROUP BY Aggregations
- SQL Query:
  SELECT s.branch, COUNT(s.student_id) as student_count, ROUND(AVG(f.amount), 2) as avg_outstanding_fee
  FROM students s
  JOIN fees f ON s.student_id = f.student_id
  WHERE f.status = 'unpaid'
  GROUP BY s.branch;

- Expected Output:
  +--------+---------------+---------------------+
  | branch | student_count | avg_outstanding_fee |
  +--------+---------------+---------------------+
  | CSE    | 12            | 67000.00            |
  | ECE    | 6             | 67000.00            |
  +--------+---------------+---------------------+


QUESTION 17: "Provide a complete attendance audit log showing the student, date, status, and the staff member who marked it."
--------------------------------------------------------------------------------
- DBMS Concept: Double Join to audit accountability
- SQL Query:
  SELECT sa.date, u.name as student_name, sa.status, staff.name as marked_by_staff, staff.role as staff_role
  FROM student_attendance sa
  JOIN students s ON sa.student_id = s.student_id
  JOIN users u ON s.user_id = u.id
  JOIN users staff ON sa.marked_by = staff.id
  ORDER BY sa.date DESC;

- Expected Output:
  +------------+--------------+---------+------------------+------------+
  | date       | student_name | status  | marked_by_staff  | staff_role |
  +------------+--------------+---------+------------------+------------+
  | 2026-05-19 | John Doe     | present | Warden           | warden     |
  +------------+--------------+---------+------------------+------------+

💡 KEY DESIGN HIGHLIGHTS TO MENTION TO YOUR TEACHER:
================================================================================
1. Composite Unique Keys: The 'student_attendance' table features a UNIQUE composite 
   key 'unique_student_date (student_id, date)'. This guarantees that no student can
   have more than one attendance record on any single calendar day, maintaining
   excellent transactional integrity.
2. Exact Currency Tracking: The 'fees' table represents money using 'DECIMAL(10,2)'
   rather than float/double. This prevents binary approximation errors and ensures
   rounding calculations strictly adhere to financial standards.
3. Domain Constraints: Table status values (like roles, leaves, and fee states) are
   strictly bounded using 'ENUM' data types. This prevents corrupt data entry 
   directly at the storage engine level.
4. Cascade Operations: 'ON DELETE CASCADE' and 'ON DELETE SET NULL' are declared on
   all Foreign Keys to automate referential cleanups when parents are modified/removed.
